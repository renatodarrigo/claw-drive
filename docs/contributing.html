Contributing

claw-drive is single-user, local, and security-sensitive. Contributions follow the same shape.

What this project values

  • The human-in-the-loop guarantee. No tool call B makes can land without going through the policy gate. Anything that weakens this is the highest-priority bug class.
  • Single-writer-per-session, file-based state. The runner is the only writer to ~/.claw-drive/sessions/<id>/. Race-free updates rely on this; don't break it.
  • The Bash PreToolUse approver is the security primitive. It runs in B's own process group, sees every tool call before it executes, and blocks until the runner answers. Replacing it with a pure-JS implementation is out of scope.
  • Tests pinned per behaviour. Every non-trivial change ships with a test that fails against the bug and passes after the fix.

In scope

  • New auto_approve / auto_reject / auto_defer patterns with positive matches AND false-positive negatives.
  • Bug fixes for races, parser edge cases, hook delivery, runner-state corruption.
  • Improvements to error messages, audit-event payloads, and observability (structured logs, event-stream filters).
  • Additive changes to the plugin: new skills, better skill prompts, clearer arg validation.
  • Documentation on the website, the README, the design spec.
  • Test coverage for any code path the existing 207-unit + 8-integration suite doesn't reach.

Out of scope

  • Multi-tenant or network-exposed flavors. claw-drive is single-user-single-host by design. PRs that add auth, remote runners, or shared state will not be accepted.
  • Anything that lets B make a tool call without going through the policy gate. Including "performance" optimisations that bypass the approver.
  • Replacing the Bash approver with a pure-JS implementation. The Bash approver runs in B's process group; that's the property the security model depends on.
  • New MCP tools, new event kinds, or breaking schema changes without a design discussion first. Open an issue describing the surface change before coding.
  • New runtime dependencies. The Bash approver requires only jq and nc. The Node side ships with one MCP-SDK dep. New deps need justification.

Before opening a PR

  1. Read docs/superpowers/specs/2026-04-21-claw-drive-design.md §invariants. If your change touches the runner, the policy evaluator, or the event log, that section is the source of truth.
  2. Run the test suite: 
    npm run test:unit          # 207 tests, no real claude, ~3s
npm run test:integration   # 8 tests, spawns real claude (costs tokens)
    Both must pass on main before you start. If they don't, file an issue first.
  3. For non-trivial changes (new MCP tool, new event kind, new policy field, new install path, default-policy edit), open an issue describing the design before writing code. Drive-by behaviour changes that ship without discussion get bounced back.

PR shape

  • Small and focused. One concern per PR. Five-line PRs are normal.
  • Branch from main. Direct-to-main is the maintainer pattern; contributors use feature branches.
  • Include a regression test. For bug fixes: a test that fails on the parent commit and passes on yours. For new patterns: a positive match AND at least one false-positive negative.
  • Conventional-ish commit prefixes match recent history: feat:, fix:, docs:, chore:, merge:. Subject under 70 chars, body explains the why.
  • Update CHANGELOG.md in the same commit if the change is user-visible. Match the existing release-note shape.

Reporting security issues

The following bug classes get a private channel — do not open a public GitHub issue:

  • Policy-evaluation bypass (B makes a tool call that should have been rejected/deferred and isn't).
  • Hook bypass (B makes a tool call without the PreToolUse hook firing).
  • Sandbox escape (B reads or writes outside its cwd in ways the policy didn't authorise).
  • Approver race or state-corruption that lets a paused call resolve incorrectly.
  • Any other failure of the human-in-the-loop guarantee.

Email renato.darrigo@gmail.com with reproducer and impact analysis. Public disclosure follows the fix.

All other bugs: open a GitHub issue.

Tone

Be precise, be patient, be kind. PRs that miss the rules above get closed politely with a pointer back here.
