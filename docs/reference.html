Used by Monitor flows. Sentinel filter is on by default — turn_completed surfaces only when the trailing token is [NEEDS-INPUT] or [DONE]. --no-token-filter disables the filter entirely. --decision-only and --only are kind-level filters that compose with the sentinel filter. --idle-after SECONDS emits a synthetic idle event after N seconds of no surfaced activity (default 600; 0 disables). See Driving patterns → Sentinel tokens.

Dynamic membership — sessions spawned later join via a periodic rescan, a stopping session closes only its own tail (its session_stopped surfaces first), and the stream runs until SIGINT. Every single-session filter applies independently per session. status (no argument) is the point-in-time fleet-snapshot companion. See Driving patterns → Watching a fleet.
provide-output <call_id> [flags]Relay human-run output to a deferred call.

Session aliases

A session can be given a human-friendly alias at start — claw-drive start --name <alias> (CLI) or start_session({ name }) (MCP) — and used in place of the canonical sess_… id at any command or tool that takes a session argument.

  • Constraints: 1–32 chars, leading letter, then letters / digits / _ / -; must not begin with sess_.
  • Uniqueness: unique among live sessions. A duplicate-live alias fails at start with an error naming the conflicting session_id; once the holder stops or is pruned, the alias is free to reuse.
  • Resolution: every session-arg CLI subcommand (send, stop, interrupt, policy, tail, show, single-session watch, single-id status, single-target pending) and MCP tool (stop_session, send_turn, poll_turn, poll_session, provide_tool_output, update_policy, interrupt_turn) accepts either an alias or a canonical id.
  • Display: status, sessions, pending, and watch --all show the alias alongside the session_id when present.

An alias lives with its session: there is no post-start rename, no cross-restart registry, and no namespacing. start still prints the canonical session_id.

Runner log

Each driven session writes <session_dir>/runner.log — the runner's own stdout/stderr (startup, stdout-loop, and teardown errors) plus Session B's stderr, captured in-process because the runner is spawned detached with stdio:"ignore". It's where to look when a Session B won't start or dies unexpectedly.

The log is bounded by size-based rotation: runner.log rolls to runner.log.1 when it would exceed LOG_MAX_BYTES (default 10 MiB), keeping up to LOG_KEEP (default 3) generations (runner.log.1 … runner.log.<KEEP>) and deleting the oldest. Override via the CLAW_DRIVE_LOG_MAX_BYTES and CLAW_DRIVE_LOG_KEEP environment variables; an absent or invalid value falls back to the default, so the bound is never disabled. Plain text only — no log levels, no time-based rotation.

Compatibility

claw-drive 1.0 is a stability promise. The versioning follows SemVer. Within the 1.x line, releases may be additive — new tools, new fields, new event kinds, new CLI subcommands or flags, new sentinel tokens — but will not remove, rename, or incompatibly change anything listed below. Breaking changes are reserved for 2.0.

The full stability contract, including the additive vs. breaking distinction and frozen field-level details, is in COMPATIBILITY.md at the repository root.
